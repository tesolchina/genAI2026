## Appendix: Key Diagrams in the Video (English Explanation)

The video contains several technical diagrams shown on screen. Below is an explanation of each, matched to the keyframe where it appears.

### Diagram 1 — Kissing number structures in low dimensions (帧 6, ~01:39)

**On screen:** "Petrie projection", "3D perspective", "rhombic triacontahedron"

**What it shows:** Visual representations of known kissing number configurations. The **rhombic triacontahedron** is the polyhedron formed by the 12 points where balls touch the centre in 3D (the answer to Newton's question). The **Petrie projection** is a way to "flatten" higher-dimensional structures into 2D drawings while preserving some symmetry. These diagrams set the stage: the problem is easy to visualise in 3D, but becomes abstract in higher dimensions.

---

### Diagram 2 — Known kissing number chart (帧 10, ~03:19)

**On screen:** Numbers 306, 240, 300, 200, 126

**What it shows:** A chart or table of known (best) kissing numbers in various dimensions. These match well-known values: K(8)=240, K(7)=126, etc. The speaker is explaining that progress has been extremely slow — only ~7 advances in 50 years — and each required a completely different mathematical method. This diagram motivates why AI was needed.

---

### Diagram 3 — Single-agent matrix completion game (帧 15, ~04:59)

**On screen:** "Filler: Matrix Completion", "Action Space"

**What it shows:** The first version of PackingStar's approach — a single agent (the "Filler") plays a matrix completion game. At each step, it selects a cosine value from the action space and fills one entry in the Gram matrix. The game ends when no more valid entries can be added. The reward = matrix size (number of spheres). This is the foundation before the two-player design was introduced.

---

### Diagram 4 — Multi-entry completion (帧 18, ~05:59)

**On screen:** "Multi-Entry Completion", "Action Space"

**What it shows:** An optimisation of the game where the agent can fill **multiple matrix entries at once** instead of one at a time. This dramatically improves trajectory efficiency — adding balls one-by-one creates many equivalent orderings (ABC = BCA), wasting exploration. Batch filling avoids this redundancy.

---

### Diagram 5 — Target kissing numbers (帧 19, ~06:19)

**On screen:** Numbers 2564, 1932, 1154

**What it shows:** The known kissing numbers the team was trying to **reproduce** before claiming new ones: K(15)=2564, K(14)=1932, K(13)=1154. Reproducing these was a key validation step — if PackingStar could not match what mathematicians had already achieved, it could not be trusted to find new results.

---

### Diagram 6 — Engineering: GPU optimisation (帧 21–24, ~06:59–07:59)

**On screen:** "Number Problem 中的挑战" (Challenges in Kissing Number Problem), "Episode 1", "Pre cached Actions"

**What it shows:** The engineering challenges and solutions. The team rewrote GPU kernels to keep data on-GPU (avoiding CPU round-trips), designed mixed-precision arithmetic, and pre-cached frequently used actions. Speaker 程远 (Yuan Cheng, Fudan/SAIS) explains that "computation and intelligence are inseparable" — faster search = smarter system.

---

### Diagram 7 — Checkpoint mechanism (帧 25, ~08:19)

**On screen:** "Without Checkpoints" vs. "With Checkpoints"; "Save state periodically. Recover from last checkpoint"

**What it shows:** A before/after diagram. Without checkpoints, a cluster crash means restarting from scratch. With checkpoints (saving state periodically), the system recovers from the last saved point. This allowed uninterrupted deep search across days, essential for scaling to 30+ dimensions.

---

### Diagram 8 — The 14D nightmare: progress chart (帧 27–28, ~08:59–09:19)

**On screen:** 2564, 1932 (帧 27); 1528, 1551, 1587 with "Kissing Number" labels (帧 28)

**What it shows:** The team's struggle to reproduce K(14)=1932. After thousands of experiments, the system could only reach ~1700 (the researcher even dreamed of 1700+). The chart likely shows successive runs climbing from 1528 → 1551 → 1587 but falling short of 1932. This "nightmare" lasted 6+ months and was only resolved by introducing the second agent (Corrector).

### Diagram 9 — PackingStar system overview (帧 31, ~10:19)

**On screen:** "PackingStar 系统"

**What it shows:** The complete system architecture. This is the top-level diagram showing how all components fit together.

### Diagram 10 — Two-Player Matrix Completion Game (帧 32–33, ~10:39–10:59)

**On screen:** "Two-Player Matrix Completion Game", "Filler", "Action Space", "Lv2"

**What it shows:** The final and key design innovation. Two agents cooperate:
- **Filler (Player 1):** Adds cosine values to the matrix (places balls).
- **Corrector (Player 2):** Removes bad entries that would "poison" the structure ("一个老鼠害一锅汤" — one bad apple spoils the barrel).

The "Lv2" label suggests a levelled/hierarchical game structure. This design was inspired by the mathematician's instinct to manually delete bad balls, and it immediately solved the 14D problem (reaching 1932). It acts as a regularisation constraint on the search space.

---

### Diagram 11 — Leech lattice: 196,560 (帧 34, ~11:19)

**On screen:** Number 196560

**What it shows:** The kissing number in 24 dimensions, K(24) = 196,560. This is the Leech lattice, one of the most famous structures in mathematics. PackingStar uses subsets of Leech lattice vectors as building blocks for dimensions 25–31. The number is highlighted to show the scale of the problem at high dimensions.

---

### Diagram 12 — Fragment Expansion (帧 39, ~12:59)

**On screen:** "Episode 3", "Fragment Expansion"

**What it shows:** The fragment decomposition mechanism. After each game round, the resulting matrix is broken into smaller sub-structures ("fragments" / 碎片). These fragments are then recombined and used as seeds for new games — "like cell division, constantly producing new digital life." It was during this process that a strange fragment in the 200th experiment led to the 25D breakthrough.
