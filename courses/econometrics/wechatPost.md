顶级计量经济学家的 Claude Code 工作流
 


最近，Emory 大学经济学教授 Pedro Sant'Anna 在个人网站上公开了他使用 Claude Code 进行科研和教学的完整工作流，同时在 GitHub 上开源了全部配置文件。Sant'Anna 是做差分法（DiD）因果推断的，Callaway & Sant'Anna (2021, JoE) 那篇处理交错 DID 的论文在应用计量圈子里几乎人人都读过，Google Scholar 引用超过 1 万次。他同时担任 Journal of Econometrics 和 JBES 的副主编。

这不是一个程序员写的技术教程，而是一位和我们背景完全相同的经济学研究者，在实际科研和教学中摸索出来的一整套方法。他用这套系统开发了一门博士课程（Econ 730: Causal Panel Data），产出了 6 套完整的讲义、800 多页 PPT 以及配套的 R 代码。整套工作流以 MIT 许可开源，任何人都可以直接拿去用。

这篇文章面向对 AI 工具不太了解的老师和同学，解读这套工作流到底在做什么，以及我们能从中借鉴什么。

先说 Claude Code 是什么
很多老师可能听说过 ChatGPT 或者 Claude 这类大语言模型，平时也在网页版的对话框里用过。但 Claude Code 是另一个东西。它是 Anthropic 公司推出的一个命令行工具，直接运行在你的电脑终端里，能够读写你本地的文件、执行代码、操作 Git 版本控制——简单说，它不是一个你跟它聊天的对话框，而是一个住在你电脑里的研究助手，直接在你的项目文件夹中干活。

举个例子：你跟它说“把这个 R 脚本里的 OLS 回归改成 IV 回归，工具变量用这两个”，它会直接打开你的 R 文件，找到相关代码，改好，保存。你跟它说“帮我把这篇论文的第三节重写，加入最新的文献讨论”，它会读取你的 LaTeX 文件，定位到第三节，直接在原文上修改。你跟它说“把这张表的格式从 AER 改成 JoE 的要求”，它也能做。

它跟网页版 AI 最大的区别在于：它直接和你的文件系统交互。不需要你把代码复制粘贴到对话框再复制回来。它就在你的项目目录里工作，跟你用同一套文件。

当然，这只是基本功能。Sant'Anna 做的事情，是在此基础上建立了一套远比问答复杂得多的系统。

Sant'Anna 建了一套什么系统
想象你雇了一个研究助理。普通的用法是：你每次给他一个小任务，他做完交给你，你检查，发现问题，退回去改，再交，再检查。这个往返过程很耗时间，而且助理不会自己主动检查质量。

Sant'Anna 的做法不同。他把 Claude Code 配置成了一个项目承包商（contractor）而不是打杂助理。你不是给它一个个小指令，而是描述你要什么结果，然后它自己完成从规划到实施到质检的全过程。

具体来说，当 Sant'Anna 给出一个任务——比如把第三讲的幻灯片从 Beamer 格式转换成 Quarto 网页格式——系统的运作方式是这样的：

首先，Claude 会进入规划模式。它不会立刻动手，而是先分析任务涉及哪些文件、可能遇到什么问题、打算分几步完成，然后把这个计划呈现给你审批。这很像你给博士生布置课题后，让他先交一个研究计划再动手一样。

你批准计划后，Claude 开始自主执行。执行完毕，它不会直接把结果交给你，而是先调用一组专门的审查员来检查自己的工作。这些审查员各司其职：一个只负责校对语法和拼写，一个只检查幻灯片的视觉排版是否溢出，一个专门评估教学质量——叙事是否连贯、数学符号是否太密集、节奏是否合理。还有一个专门审查 R 代码的质量——随机数种子有没有设、路径有没有硬编码、代码能不能复现。

如果审查员发现了问题，系统会自动修复并重新审查，最多可以来回五轮。每个文件最终会得到一个 0 到 100 的质量分数，80 分以下不允许提交到版本库。

这整个过程中你做的事情就是：描述任务、批准计划、最后检查结果。中间环节全部自动化。

这跟实证研究有什么关系
可能有老师会觉得：这跟我有什么关系？关系在于 Sant'Anna 解决的那个核心问题：如何在使用 AI 工具的同时，维持学术研究需要的质量标准。

做实证研究的人都知道，一个项目到后期文件会变得很多：论文的 LaTeX 源文件、数据清洗脚本、回归分析脚本、robustness check 脚本、图表生成脚本、数据描述文档，再加上对应的输出文件。这些文件之间存在复杂的依赖关系——改了模型设定，所有下游的图表和文字描述都要跟着改。靠人工维护一致性极其痛苦，这也是为什么很多论文到了 revise 阶段会出现图文不符的 bug。

Sant'Anna 的方案里有一个原则叫单一真相源——对于任何一块内容，只有一个权威版本，其他所有版本都从这个源头派生。这个思路直接适用于我们的实证研究：回归表格的数字只在 R/Stata 脚本的输出中定义一次，论文中的表格从输出自动生成，幻灯片中的同一张表也从同一个输出派生。任何修改只需改一处，下游全部自动更新。

还有一个设计是复现协议。扩展模型之前，系统会先自动复现原始结果，并设好数值容差——点估计容差 1e-6、标准误容差 1e-4、覆盖率容差 ±0.01。数字对不上就提醒你可能有 bug。做 Monte Carlo 或者 bootstrap 的人都知道，这种核对工作量本来很大，现在可以自动化掉。

他还设了一个探索文件夹，专门放试探性的分析——比如你想试一个新的工具变量、换一种估计方法、或者跑一组 pilot simulation。放进探索文件夹的代码质量门槛降低到 60 分（而不是正式代码的 80 分），不需要走完整的审查流程，可以快速试错。一旦实验成功，再正式升级到生产文件夹并接受完整质检。这个思路很实用：研究的探索阶段和正式阶段确实需要不同的严格程度，但很多人容易在两个阶段之间混淆——要么探索阶段太拘谨导致不敢试，要么正式阶段太随意导致出错。

对教学意味着什么
Sant'Anna 自己开发这套系统的直接动机就是教学——他用它制作了一整门博士课程的讲义。他配置了一个教学质量审查员，这个审查员会从 13 个维度评估一套幻灯片：概念的引入是否遵循从直觉到形式化的渐进路径？每张幻灯片的信息密度是否合理？整个讲次是否有清晰的叙事弧线？符号是否全程一致？有没有在没有铺垫的情况下突然引入新符号？

做课件多的老师都知道，很多问题自己反复看也看不出来。连续盯着幻灯片几个小时后，对细节基本已经麻木了——这时候一个不会累的审查员就派上用场了。

Sant'Anna 还做了一套对抗式审查机制：一个 AI 角色（critic）专门负责挑毛病，另一个 AI 角色（fixer）专门负责修复。两者严格分工，挑毛病的不能自己改，改的不能自己评判改得好不好。然后让 critic 再审查 fixer 的修复，如此往返，最多五轮。道理很简单：一个 agent 同时负责挑毛病和改毛病，容易自我满足——毕竟修的是自己发现的问题，很难客观。拆成两个对立角色，质量就上去了。

几个有意思的设计细节
Sant'Anna 系统里有几个设计细节，拿出来说说。

第一个是持续学习机制。每次 Claude 犯了错——比如把 Belloni (2013) 引用成 Belloni (2014)，或者该用 T_i 的地方写成了 T_t——你纠正之后，这条纠正会保存下来。下次开新会话时，Claude 先读一遍这些历史纠正，避免重蹈覆辙。说白了就是给它建了个错题本。积累一段时间后，Claude 对你领域的符号惯例、引用习惯、代码风格会越来越熟。

第二个是他把 CLAUDE.md（给 AI 的项目说明书）控制在合理长度，具体规则、技巧、审查标准拆成十几个独立文件。好处是模块化——想改审查标准就只动那一个文件，不用在一个巨大文档里翻找。整套配置也因此可以版本管理、跨项目复用。

第三个是他对什么时候该严格、什么时候该宽松的区分。生产代码 80 分才能提交，探索代码 60 分就行。正式合并时才生成质量报告，日常提交不做这个开销。这种分层的严格度设计避免了要么彻底不管、要么事事审查的两难。

跟我们有什么关系
Sant'Anna 这套工作流，重点不是说你也应该搞这么复杂的系统。重点是：现在的 AI 工具能做的事，已经远不止帮你润色文字或解释代码了。配置得当的话，它能接手实证研究中大量结构化但繁琐的活——代码质检、文档一致性维护、格式转换、复现验证——而且质量过得了期刊审稿。

跟我们直接相关的有几个方面：

科研上，我们大多数人手里同时有好几个项目，每个项目几十个代码文件和文档。维护它们之间的一致性、确保 revise 时不引入新 bug，本身就耗精力。像 Claude Code 这样能直接操作文件的工具，可以把这部分活接过去。

教学上，课件制作和更新占了不少老师的时间。AI 能帮的不只是生成内容——Sant'Anna 的系统更侧重审查：检查已有课件里的不一致、排版问题、叙事断裂。这种质量保障可能比内容生成更有用，因为我们通常不缺内容，缺的是盯着几百页课件保持高质量的精力。

研究生培养上，有个绕不开的现实：学生已经在大量用 AI 了。与其回避，不如想想怎么引导他们用好。Sant'Anna 这套系统里的质量门控和复现协议，其实是用系统设计来兜底研究质量——AI 不光用来干活，还用来检查活干得对不对。这个思路对带学生挺有启发。

如何进一步了解
Sant'Anna 的博客文章和 GitHub 仓库都是公开的。仓库做成了即拿即用的模板——领域专属内容都换成了占位符，fork 一份填上自己的东西就能用。

如果有兴趣但对 Claude Code 完全不熟悉，我建议的路径是：先装一个 Claude Code 试试基本功能（读写文件、执行代码），感受一下它跟网页版 AI 的区别。有了这个基础再看 Sant'Anna 的系统，会容易理解得多。
